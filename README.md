Event-driven data processing engine designed to monitor file activity, enrich events with semantic information, and generate structured datasets for behavioral and productivity analysis.

The system continuously observes file activity on a machine and transforms low-level filesystem events into structured analytical records stored in a data lake.

---

# Architecture

The engine follows a **Flywheel Architecture** designed for stability and scalable event processing.

Watcher → EventBus → WorkerPool → Enricher → Data Lake → Analytics


Each component is isolated and communicates through structured records, allowing the system to evolve into larger data pipelines.

---

# Core Concepts

## DataRecord

Atomic data unit that flows through the pipeline.

Each event detected by the watcher becomes a structured record containing:

- timestamp
- event type
- file metadata
- extension
- file hash
- contextual temporal features

Example fields generated by the record object: :contentReference[oaicite:0]{index=0}

- timestamp
- event_type
- arquivo
- pasta
- extensao
- hash
- tamanho
- hour
- weekday

This structure makes the system immediately usable for machine learning datasets.

# Event System

The engine uses an event contract to standardize system signals.

Supported events include: :contentReference[oaicite:1]{index=1}

- FILE_CREATED
- FILE_MODIFIED
- FILE_DELETED
- FILE_COPIED
- QUEUE_HIGH
- QUEUE_LOW

File extensions are mapped to **human intent categories**, such as:

- writing
- analysis
- development
- communication
- media_consumption
- file_management

This allows behavioral interpretation of raw file activity.

# Processing Pipeline

The processing pipeline is implemented inside the main engine loop.

Flow:

1. Watch filesystem events
2. Convert to structured DataRecord
3. Push into WorkerPool
4. Enrich semantic metadata
5. Deduplicate using file hash
6. Persist event into data lake
7. Update daily summaries

Key responsibilities include:

### File stabilization

The system waits until the file is fully written before processing.

### Hash deduplication

MD5 hashes are used to identify duplicate files and detect copy events.

### Session tracking

Events are grouped into behavioral sessions before entering the data lake.

### Semantic enrichment

The system infers:

- user intent
- action weight
- time-of-day context

# Worker Architecture

Processing concurrency is handled by a worker pool. :contentReference[oaicite:2]{index=2}

The pool:

- consumes records from a queue
- processes them in parallel threads
- emits backpressure signals if the queue grows too large

Backpressure protection is triggered when the queue reaches a high watermark.

This prevents uncontrolled memory growth and signals load conditions to the system.
